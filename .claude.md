## CONTEXTO

Quero construir um sistema web para automatizar a **relotação** dos ~300 procuradores da Procuradoria-Geral do Estado do Rio de Janeiro (PGE-RJ). Relotação é o processo periódico de redistribuir os procuradores entre as áreas da instituição.

Já existe um protótipo de front-end em HTML/CSS/JS (apenas visual). Agora preciso da implementação real, com persistência de dados e a lógica de negócio.

**Stack desejada:** _(preencha — ex: back-end em Python/FastAPI + PostgreSQL)_

## DOMÍNIO

São 30 áreas (configurável):
- **16 Especializadas** (PG-03 … PG-19)
- **13 Regionais e Postos Avançados** (1PR-NIT … 10PA-CG)
- **1 Gabinete** (PG-02)

Cada área tem vagas de até 5 tipos, identificadas por cor, que diferem em **quem decide** o ocupante:

| Cor | Tipo | Quem preenche | Onde existe |
|---|---|---|---|
| Cinza | PG | Procurador-Geral (cargo único) | Só gabinete |
| Vermelho | Nomeação | Gestão (livre nomeação) | Todas |
| Verde | Escolha do Chefe | Chefe da área indica | Especializadas |
| Amarelo | Designação do PG | Procurador-Geral designa | Só gabinete |
| Azul | Acervo | **Sistema preenche automaticamente** | Todas |

// Procurador
interface Procurador {
  id: number;
  nome: string;
  antiguidade: number;        // 1..N, único e imutável no ciclo. 1 = mais antigo
  status: 'LOTADO' | 'PENDENTE' | 'EM_LICENCA' | 'VACANCIA';
  lotacaoAtualCodigo: string | null;   // null se vacância
  ativo: boolean;             // false se EM_LICENCA ou VACANCIA
  // relações
  preferencias: Preferencia[];
  historico: Lotacao[];
}

// Área
interface Area {
  codigo: string;             // PK: "PG-04", "1PR-NIT", "PG-02"
  nome: string;
  tipo: 'ESPECIALIZADA' | 'REGIONAL' | 'GABINETE';
  vagasPG: number;            // cinza (0 ou 1, só gabinete)
  vagasNomeacao: number;      // vermelho
  vagasEscolha: number;       // verde
  vagasDesignacao: number;    // amarelo (só gabinete)
  vagasAcervo: number;        // azul
  rotulosNomeacao?: string[]; // ["Chefe", "Assessor", ...]
  rotulosDesignacao?: string[];
  // derivado: totalVagas = soma dos 5 contadores
}

// Vaga (instância de posição numa área)
interface Vaga {
  id: number;
  areaCodigo: string;
  numero: number;             // posição na área (1,2,3...)
  tipo: 'PG' | 'NOMEACAO' | 'ESCOLHA' | 'DESIGNACAO' | 'ACERVO';
  cargo?: string;             // rótulo do cargo
  ocupanteId: number | null;  // null = vaga aberta
  origem: 'AUTOMATICA' | 'MANUAL';
}

// Preferência (ranking de um procurador por uma área)
interface Preferencia {
  procuradorId: number;
  areaCodigo: string;
  ordem: number;              // 1 = mais querida
}

// Lotação (registro histórico)
interface Lotacao {
  id: number;
  procuradorId: number;
  areaCodigo: string;
  dataEntrada: string;        // ISO date
  dataSaida: string | null;   // null = atual
  motivo: 'POSSE_INICIAL' | 'NOMEACAO' | 'ESCOLHA_CHEFE' | 'DESIGNACAO_PG'
        | 'ACERVO' | 'PERMANENCIA' | 'SUBSTITUICAO_MANUAL';
  cicloId: string;
}

// Ciclo
interface Ciclo {
  id: string;                 // "2026.1"
  abertura: string;
  encerramento: string | null;
  status: 'EM_CURSO' | 'ENCERRADO' | 'CANCELADO';
  snapshot?: Vaga[];          // congelado ao encerrar
  totalProcuradores: number;
  totalVagas: number;
  movimentacoes: number;
  permanencias: number;
  pctPrimeiraPref: number;
}

## REGRAS DE NEGÓCIO

### R1 — Ordem de preenchimento
1. Vermelhas (Nomeação) → gestão manual
2. Amarelas (Designação PG) → gestão manual
3. Verdes (Escolha do chefe) → registradas pela coordenação
4. Azuis (Acervo) → **automático**, após R2

### R2 — Algoritmo de Acervo (Serial Dictatorship por antiguidade)
**Esta é a regra central.** Percorrer os procuradores em ordem de antiguidade crescente (mais antigo primeiro); cada um ocupa a primeira área da sua lista de preferências que ainda tenha vaga azul livre.

```python
def alocar_acervo(procuradores, vagas):
    # pool inclui vagas liberadas por licença e vacância
    disponiveis = [v for v in vagas if v.tipo == 'ACERVO' and v.ocupante is None]
    for proc in sorted(procuradores, key=lambda p: p.antiguidade):
        if not proc.ativo:
            continue   # pula EM_LICENCA e VACANCIA
        for pref in sorted(proc.preferencias, key=lambda x: x.ordem):
            vaga = next((v for v in disponiveis
                         if v.area_codigo == pref.area_codigo and v.ocupante is None), None)
            if vaga:
                vaga.ocupante = proc.id
                vaga.origem = 'AUTOMATICA'
                break
```

Invariantes:
- Antiguidade menor escolhe antes (ordem fixa, sem disputa entre procuradores).
- Preferência só é comparada dentro da lista do próprio procurador.
- Inativos não ocupam vaga, mas suas vagas antigas entram no pool.

### R3 — Edição manual sempre permitida
A gestão pode trocar/remover qualquer ocupante a qualquer momento, inclusive azul. Ao editar um azul, `origem = MANUAL`. Não há homologação que trave edições.

### R4 — Controle de orçamento
Σ totalVagas de todas as áreas ≤ total de procuradores (300). Avisar visualmente excedente/folga.

### R5 — Licença libera vaga
Procurador `EM_LICENCA` não participa; sua vaga vai ao pool de acervo. Ao voltar, reentra no próximo ciclo conforme antiguidade.

### R6 — Vacância libera vaga
`VACANCIA` (aposentadoria/falecimento) também libera a vaga para o pool, igual à licença. A vacância permanece registrada com o nome do antigo titular até ser preenchida.

### R7 — Ciclos
Apenas um ciclo `EM_CURSO` por vez. Encerrar: congela snapshot, gera `Lotacao` por procurador (entrada/saída), gera ato administrativo (PDF). Encerrar NÃO impede edições futuras (que entram no próximo ciclo).
